# Training Layer

!!! abstract "Layer at a glance"
    **Receives:** `list[FeatureVector]` (from feature layer output files) + researcher-supplied `labels.csv`
    **Produces:** saved model directory (written by `model.save()`)
    **Protocol:** `Trainer` → `fit(dataset)`, `save(path)`
    **Files:** `training/base.py` · `training/dataset.py` · `training/trainer.py`
    **Parallel to:** [Inference layer](inference.md) — shares `FeatureVector` input and `MLModel` contract, but neither imports from the other.
    **Guide:** [Preparing Labels](../guides/label-preparation.md)

The training layer loads labeled features, splits them into train/val/test sets, and
fits a model. It is parallel to inference — neither layer imports from the other.

```
src/ml4em/training/
  base.py         Trainer Protocol
  dataset.py      FeatureDataset — load features + join labels
  trainer.py      StandardTrainer
```

---

## How the pieces connect

```text
FeatureDataset.from_storage(cfg.storage, labels_path)
  ├─ loads parquet files (written by FeaturePipeline)  → list[FeatureVector]
  └─ joins labels.csv on source_id                     → list[LabeledSample]

dataset.split(val_fraction=0.1, test_fraction=0.1, seed=42)
  └─→ (train, val, test)  subsets of LabeledSample

StandardTrainer(model, cfg.training)
  ├─→ fit(dataset)   → model.fit(train, val)  [stub — NotImplementedError]
  └─→ save(path)     → model.save(path)        [implemented]
```

**Entry points:** `FeatureDataset.from_storage` to load data; `StandardTrainer.fit` to train; `StandardTrainer.save` to persist the result.

---

## Protocol — `Trainer`

```python
class Trainer(Protocol):
    def fit(self, dataset: FeatureDataset) -> None: ...
    def save(self, path: str) -> None: ...
```

---

## `FeatureDataset`

Loads `FeatureVector` objects from parquet files written by `FeaturePipeline` and
joins them with a researcher-supplied labels CSV.

### Labels CSV format

```csv
source_id,label
686149073900013696,1
686149073900013697,0
686149073900013698,0
```

- `source_id`: the same string ID used in `LightCurve.source_id`
- `label`: `1` = positive class (what you're looking for), `0` = background

Labels are **never generated by ml4em**. They must come from an external source — for
example, a cross-match against a spectroscopic white dwarf catalog. See
[Preparing Labels](../guides/label-preparation.md).

Sources present in only one of (features, labels) are silently skipped.

### Loading and splitting

```python
from ml4em.training import FeatureDataset

dataset = FeatureDataset.from_storage(cfg.storage, labels_path="labels.csv")

# Inspect class balance
print(dataset.class_counts())       # {0: 5000, 1: 320}
print(dataset.positive_fraction())  # 0.060

# Split into train / val / test
train, val, test = dataset.split(val_fraction=0.1, test_fraction=0.1, seed=42)
```

### Class imbalance

Rare astrophysical objects (like white dwarf binaries) are genuinely rare. In a typical
field sample, the positive class might be 1–5% of all sources.

```python
dataset.positive_fraction()  # → 0.04  (4% positive)
```

This imbalance affects training: a model that predicts "background" for everything
would be 96% accurate but completely useless. Handling class imbalance is left to the
model implementation (XGBoost has a `scale_pos_weight` parameter for this; neural nets
can use class-weighted loss).

## `StandardTrainer` *(shell)*

```python
from ml4em.training import StandardTrainer

trainer = StandardTrainer(model, cfg.training)
trainer.fit(dataset)   # → NotImplementedError (training loop pending)
trainer.save(path)     # delegates to model.save(path) — implemented
```

`cfg.training` controls loop parameters:

| Parameter | Description |
|-----------|-------------|
| `learning_rate` | Step size for gradient descent (used by neural net trainers) |
| `batch_size` | Samples per gradient update |
| `max_epochs` | Maximum training epochs |
| `patience` | Early stopping: stop after this many epochs with no val improvement |
| `val_fraction` | Fraction of data held out for validation |
| `test_fraction` | Fraction of data held out for final evaluation |
| `seed` | Random seed for reproducibility |

!!! note "Status"
    The training loop (`fit`) is a stub raising `NotImplementedError`. `save` works
    and delegates to `model.save(path)`. The training loop will be implemented once
    the model architecture is finalized.

    XGBoost training may not need all of these parameters (`max_epochs` doesn't apply
    to XGBoost in the traditional sense; early stopping uses `eval_set` instead).
    The `StandardTrainer` will dispatch appropriately per model type.
